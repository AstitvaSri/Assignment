1 Objective

We look at how to map a data model to a schema for tables, both manually, and with the use of the SQL Alchemy Object Relational Mapping library in Python.

2 Tables

A table is a structure with rows and columns. In each row, an entry in a column may be empty.

Now we may think of the table as a set of rows. And the columns as defining relations.

Rows    = {1, 2, 3, 4, 5, 6} 
Names   = {a, b, c}  the set of values in the name column

Names^  = Names union {NULL} 
Vals^   = Vals union {NULL}

name and val are functions from Rows to Names^ and Vals^. 

name = def =   {(1, a), (2, b), (3, c), (4, a), (5, NULL),(6, a)}

name: Rows+    >  Names^!
name: Rows+    >  Names?
val:  Rows+    >  Vals^!
val:  Rows+    >  Vals?

3.3 User

3.3.1 Schema and Foreign Keys

|-----+----+---------+----------|
| row | id | name_id | email_id |
|-----+----+---------+----------|


|-------------+-------------+-------------|
| id          | name_id     | email_id    |
|-------------+-------------+-------------|
| Int         | Int         | Int         |
|-------------+-------------+-------------|
| Primary Key | Foreign Key | Foreign Key |
|-------------+-------------+-------------|
  • A column labelled as a foreign key means that the value in that column is a primary key of another table.
  • Since databases are flat (entities can't nest other entities), some entries are pointers to other entities.
  • This is done by employing keys of those other entities. Such keys play the role of a foreign key.

3.3.2 Summary with example

row id name_id email_id
  Int Int Int
  Primary not nullable not nullable
      unique
1 295 702 394
2 631 495 207

3.3.3 MySQL

mysql> desc user;
+----------+---------+------+-----+---------+----------------+
| Field    | Type    | Null | Key | Default | Extra          |
+----------+---------+------+-----+---------+----------------+
| id       | int(11) | NO   | PRI | NULL    | auto_increment |
| name_id  | int(11) | NO   | MUL | NULL    |                |
| email_id | int(11) | NO   | UNI | NULL    |                |
+----------+---------+------+-----+---------+----------------+

The MUL attribute refers to the fact that the name_id relation is not injective: it is possible for more than one user row to have the same name, i.e., name_id: Rows+ > Name_Id!

4 ORM

  • So far, we started with data model and mapped the data model to the tables.
  • An ORM is a library that lets us programmatically specify how to construct the tables.
  • It does this by defining a class and its fields, and how each field maps to the databases's columns or relations.

Let's look the mapping between the tables and the objects.

4.1 Email

args = {"__tablename__": "email_tbl",
        "id": db.Column(db.Integer, primary_key=True),
        "val": db.Column(db.String(128), unique=True, nullable=False),
        }

Email = ClassPersistenceTemplate.mk_persistent(Email, ['val'], **args)

Let us look at each line in the above:

4.1.1 tablename

"__tablename__": "email_tbl",

This specifies email_tbl as the name of the table in the database that corresponds to the class Email.

4.1.2 id

    "id": db.Column(db.Integer, primary_key=True),

This specifies the field id as the table's id column. The attributes (here, primary_key) are also specified. Note how this corresponds to the mathematical specification id: Rows! > Ids!, which in turn means an isomorphism between Rows and Ids.

4.1.3 val

    "val": db.Column(db.String(128), unique=True, nullable=False),

This specifies the field val to be the table's val column. The attributes (here, unique and nullable) are also specified. Note how this corresponds to val: Rows! > Vals!.

4.1.4 Persisting a class with mk_persistent

Email = ClassPersistenceTemplate.mk_persistent(Email, ['val'], **args)
  • This line specifies that Email is now redefined to be a class with persistence.
  • ['val'] specifies the (possibly compound) key. This information is used to locate the objects in the corresponding table.
  • **args are the specification to create the class Email

4.2 Name

args = {"__tablename__": "name_tbl",

Name = ClassPersistenceTemplate.mk_persistent(Name, ['val'], **args)

Let us look at each line in the above:

4.2.1 tablename

"__tablename__": "name_tbl",

This specifies name_tbl as the name of the table in the database that corresponds to the class Name.

4.2.2 id

    "id": db.Column(db.Integer, primary_key=True),

This specifies the field id as the table's id column. The attributes (here, primary_key) are also specified.

4.2.3 val

    "val": db.Column(db.String(128), unique=True, nullable=False),

This specifies the field val to be the table's val column. The attributes (here, unique and nullable) are also specified.

4.2.4 users

The line 

        "users": db.relationship('User', backref='name')

creates two relations, each inverse of the other: 

users: name_tbl > user-tbl 
name: user_tbl >  name_tbl

Operationally, the python statement above does the following:

  1. It defines a field called users in a Name object

    1. which specifies the relationship users: name_tbl > user_tbl.
    2. This relationship users takes a name, say

    n:name_tbl and computes all users with that name as follows:

    1. Go to the table specified in User class (i.e., user_tbl).

      1. In the table (user_tbl) find the foreign key

      column that refers to the the table of this class. (See definition of User)

      1. The class is Name. Its table is name_tbl. The foreign key is name_tbl.id
      2. Find all users whose foreign key is equal to the id field of n.
  2. It creates an object field name in the User class. This field computes the inverse of users: given a user, it computes the name of the user.

See http://docs.sqlalchemy.org/en/latest/orm/backref.html for a detailed explanation of backref.

4.2.5 Creating persistent class

Name = ClassPersistenceTemplate.mk_persistent(Name, ['val'], **args)
  • This line specifies that Name is now redefined to be a class with persistence.
  • ['val'] specifies the (possibly compound) key. This information is locate the objects in the corresponding table.
  • **args are the specification to create the class Name

4.3 User

4.3.1 User roles

users_roles = db.Table('users_roles',
                      db.Column('user_id',db.Integer, db.ForeignKey('user_tbl.id')),
                      db.Column('role_id',db.Integer,db.ForeignKey('role_tbl.id')))

Creates a table users_roles that stores the relation users_roles:user_tbl > role_tbl.

4.3.2 User

args = {"__tablename__": "user_tbl",
        "id": db.Column(db.Integer, primary_key=True),
        "name_id": db.Column(db.Integer, db.ForeignKey('name_tbl.id'), 
                                 nullable=False, unique=False),
        "email_id": db.Column(db.Integer, db.ForeignKey('email_tbl.id'),
                                 nullable=False, unique=True),
        "roles": db.relationship('Role',
                                     secondary=users_roles, backref='users'),
        "email": db.relationship('Email',
                                    backref=db.backref('user', uselist=False))
        }

User = ClassPersistenceTemplate.mk_persistent(User, ['email'], **args)
  • Again, note the columns name_id, email_id, whose values are foreign keys, i.e., ids from other ('foreign') tables.
  • roles and users define a pair of inverse relations between usr_tbl and roles_tbl (not shown). They are computed and available as fields in their respective objects.
  • email and user define a pair of inverse relations between user_tbl and email_tbl. They are computed and made available as fields in their respective objects.
  • Finally, the field email acts as a key.
